# 源级充电视频重试设计

## 目标

增加一个手动的 UP 投稿源级操作，用来重试某个 UP 投稿源里已经识别到的充电视频。这个功能不做定期全源扫描，也不在按钮点击时预先检查是否已经充电。按钮只负责把当前 UP 投稿源里的充电视频重新放回现有下载流程。

## 范围

- 只在 UP 投稿源卡片上增加一个按钮：`重试充电视频`。
- 新增后端接口：
  `POST /api/video-sources/{source_type}/{id}/retry-charge-videos`。
- 接口只接受 `source_type = submission`，其他源类型返回不支持该操作。
- 只影响当前选中的 UP 投稿源。
- 只影响 `is_charge_video = true` 的视频。
- 保留现有下载行为：
  - 如果账号现在已经有权限播放该充电视频，正常下载逻辑会下载真实媒体文件；
  - 如果账号仍然没有权限播放，现有占位逻辑继续保留或重新创建占位文件。

## 后端设计

接口先校验 `{source_type, id}`。只有 `source_type = submission` 时继续处理；收藏夹、合集、稍后再看、番剧等源类型直接返回“不支持该操作”的 API 错误。

接口根据投稿源 ID 复用现有投稿源过滤逻辑，找到当前 UP 投稿源下满足以下条件的视频：

- `valid = true`
- `deleted = 0`
- `auto_download = true`
- `is_charge_video = true`

对匹配到的视频，只重置“重新尝试媒体下载”需要的状态：

- 视频级的“分页下载”状态重置为待执行；
- 分页级的“视频内容下载”状态重置为待执行；
- `auto_download` 保持为 true；
- `valid` 保持为 true。

接口返回：

- 是否有内容被重置；
- 重置了多少个视频；
- 重置了多少个分页；
- 一条给前端展示的消息。

如果至少重置了一个视频或分页，接口只返回重置结果，不主动触发扫描。后续由下一轮定时扫描，或用户手动点击现有“任务刷新”来让下载管线处理这些重试。

## 前端设计

视频源页面只在 UP 投稿源卡片操作区增加一个紧凑按钮，沿用现有按钮样式和 lucide 图标模式。点击后：

- 调用新的 API 方法；
- 请求期间只禁用当前源的该按钮；
- 成功后用 toast 显示重置数量；
- 失败时沿用现有请求错误提示流程。

按钮只对 UP 投稿源可见。如果后端返回重置数量为 0，toast 显示“该 UP 投稿源没有可重试的充电视频”。如果有内容被重置，toast 说明这些视频会在下一轮扫描中重试。

## 错误处理

- 未知源类型、非投稿源类型或投稿源不存在时，返回与现有视频源接口一致的 API 错误。
- 如果当前正在扫描，接口仍然只负责完成状态重置；当前这轮扫描是否能立刻捡到这些重试项由现有扫描时序决定。接口本身不唤醒扫描循环，也不清空投稿源的 `next_scan_at`。
- 接口不删除占位文件。是否覆盖占位文件由现有下载逻辑决定。

## 测试

后端测试覆盖：

- 只影响当前选中的 UP 投稿源；
- 同一 UP 投稿源里的普通视频不变；
- 其他 UP 投稿源里的充电视频不变；
- 非投稿源调用接口会返回不支持；
- 视频级“分页下载”状态和分页级“视频内容下载”状态会被重置；
- 接口返回的计数正确。

前端验证覆盖：

- UP 投稿源卡片操作区会显示按钮；
- 非 UP 投稿源卡片不显示按钮；
- 点击按钮会调用新的 API 方法；
- 成功和零重置数量都有明确提示。
